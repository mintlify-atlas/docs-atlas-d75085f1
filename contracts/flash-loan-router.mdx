## Overview

The `FlashLoanRouter` is a solver contract for CoW Protocol that enables solvers to request flash loans before executing settlements. It acts as the central coordinator, managing the flash loan lifecycle and ensuring proper settlement execution.

### Key Features

- **Multi-Protocol Support**: Works with any flash loan provider through dedicated borrower contracts
- **Sequential Loan Processing**: Handles multiple flash loans in sequence before settlement
- **Solver Integration**: Designed to be called by authorized CoW Protocol solvers
- **Reentrancy Protection**: Uses state flags to prevent concurrent settlements
- **Settlement Validation**: Ensures only valid `settle()` calls are executed

### Contract Architecture

The router coordinates three main actors:
1. **Solvers**: Authorized addresses that initiate settlements with flash loans
2. **Borrower Contracts**: Protocol-specific adapters (Aave, ERC-3156, etc.)
3. **Settlement Contract**: The CoW Protocol settlement contract

### Immutable State Variables

After construction, the following values are set:
- `settlementContract`: The settlement contract address
- `settlementAuthentication`: The authenticator contract (retrieved from settlement contract)

## Core Functions

### flashLoanAndSettle

```solidity
function flashLoanAndSettle(
    Loan.Data[] calldata loans,
    bytes calldata settlement
) external onlySolver
```

Main entry point for solvers to request flash loans and execute a settlement.

<ParamField path="loans" type="Loan.Data[]" required>
  Array of flash loan requests to be executed sequentially. Each loan specifies:
  - `amount`: Amount of tokens to borrow
  - `borrower`: Borrower contract to use (protocol-specific)
  - `lender`: Address of the flash loan provider
  - `token`: ERC-20 token to borrow
</ParamField>

<ParamField path="settlement" type="bytes" required>
  ABI-encoded call data for the settlement contract's `settle()` function (as generated by `abi.encodeCall`)
</ParamField>

**Behavior:**
1. Validates that no other settlement is in progress
2. Emits `Settlement` event with the solver's address
3. Processes each flash loan sequentially through borrower contracts
4. After all loans are obtained, executes the settlement
5. Reverts if settlement fails or state is inconsistent

**Requirements:**
- Caller must be an authorized CoW Protocol solver
- No concurrent settlement can be in progress
- Settlement must complete successfully

### borrowerCallBack

```solidity
function borrowerCallBack(
    bytes memory loansWithSettlement
) external onlyPendingBorrower
```

Callback function invoked by borrower contracts after receiving flash loan proceeds.

<ParamField path="loansWithSettlement" type="bytes" required>
  Encoded data containing remaining loans and settlement information, passed through from the borrower
</ParamField>

**Behavior:**
- Validates the callback is from the expected borrower
- Verifies data integrity by comparing hash
- Continues processing remaining loans or executes settlement

**Requirements:**
- Can only be called by the current pending borrower contract
- Data hash must match the expected value

## Modifiers

### onlySolver

```solidity
modifier onlySolver()
```

Restricts function access to authorized CoW Protocol solvers only. Validates caller against `settlementAuthentication.isSolver()`.

### onlyPendingBorrower

```solidity
modifier onlyPendingBorrower()
```

Restricts function access to the currently expected borrower contract during flash loan processing.

## State Management

The contract uses transient storage variables to manage execution flow:

- `pendingBorrower`: Tracks the current execution state
  - `READY` (address(0)): No settlement in progress
  - `SETTLING`: Flash loans complete, settlement executing
  - Borrower address: Waiting for callback from specific borrower

- `pendingDataHash`: Hash of data expected from borrower callback for validation

## Usage Example

```solidity
// Setup
FlashLoanRouter router = FlashLoanRouter(routerAddress);
AaveBorrower aaveBorrower = AaveBorrower(borrowerAddress);

// Prepare flash loan request
Loan.Data[] memory loans = new Loan.Data[](1);
loans[0] = Loan.Data({
    amount: 1000e18,              // 1000 tokens
    borrower: IBorrower(address(aaveBorrower)),
    lender: aavePoolAddress,
    token: IERC20(usdcAddress)
});

// Prepare settlement call data
bytes memory settlement = abi.encodeCall(
    ICowSettlement.settle,
    (
        tokens,
        clearingPrices,
        trades,
        interactions
    )
);

// Execute flash loan and settlement (as authorized solver)
router.flashLoanAndSettle(loans, settlement);
```

## Execution Flow

1. **Initiation**: Solver calls `flashLoanAndSettle()` with loans and settlement data
2. **Loan Processing**: For each loan:
   - Router calls borrower's `flashLoanAndCallBack()`
   - Borrower requests flash loan from lender
   - Lender transfers funds and calls borrower's callback
   - Borrower calls router's `borrowerCallBack()`
   - Router processes next loan or proceeds to settlement
3. **Settlement**: After all loans obtained:
   - Router validates settlement call data
   - Executes settlement on CoW Protocol
   - Flash loan repayments occur during settlement execution
4. **Completion**: State resets to `READY` for next settlement

## Security Considerations

### Reentrancy Protection
The contract uses state flags (`pendingBorrower`) to prevent concurrent settlements and unauthorized callbacks.

### Settlement Validation
- Only the `settle()` function selector is allowed in settlement call data
- Settlement must complete successfully (no silent failures)
- Data integrity verified through hash validation

### Access Control
- Only authorized CoW Protocol solvers can initiate settlements
- Borrower callbacks must come from expected addresses
- Settlement contract is immutable after construction

### Flash Loan Risks
- Solvers responsible for correct loan specification
- Settlement must handle flash loan repayments correctly
- Different protocols have different repayment mechanisms

<Warning>
  The router does not validate that flash loan proceeds are available for the settlement. Solvers must ensure the borrowed funds are accessible and sufficient for the intended trades.
</Warning>

## Integration Notes

### For Solvers
- Must be registered as a solver in the settlement authenticator
- Should validate loan parameters before calling router
- Settlement logic must handle flash loan repayments
- Consider gas costs of multiple sequential flash loans

### For Borrower Contracts
- Must implement the `IBorrower` interface
- Must pass data through callbacks without modification
- Should handle protocol-specific flash loan mechanics
- Must approve router/settlement for token transfers

### Flash Loan Providers
- Aave: Use `AaveBorrower` contract
- ERC-3156 compatible: Use `ERC3156Borrower` contract
- Custom protocols: Implement new borrower contract
